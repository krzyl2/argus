      1. `actions/checkout@v4`.
      2. `actions/setup-dotnet@v4` with `dotnet-version: '8.0.x'` (RESEARCH A5 — guard against runner not having .NET 8; low cost).
      3. dotnet publish step (RESEARCH Pattern 2 — MUST run before docker build, the Dockerfile is NOT multi-stage and COPYs pre-built output):
         ```yaml
         - name: Publish orchestrator
           run: >-
             dotnet publish orchestrator/Argus.Orchestrator/Argus.Orchestrator.csproj
             -c Release --self-contained false -o orchestrator/publish/
         ```
      4. `docker/setup-qemu-action@v3` (enables linux/arm64 binfmt — required for QEMU emulation per locked decision).
      5. `docker/setup-buildx-action@v3`.
      6. `docker/login-action@v3` with `registry: ghcr.io`, `username: ${{ github.actor }}`, `password: ${{ secrets.GITHUB_TOKEN }}`.
      7. `docker/build-push-action@v6` (RESEARCH Pattern 1 — single-step multi-arch, NOT the deprecated home-assistant/builder action):
         ```yaml
         - name: Build and push multi-arch image
           uses: docker/build-push-action@v6
           with:
             context: .
             file: argus/Dockerfile
             platforms: linux/amd64,linux/arm64
             push: true
             tags: |
               ghcr.io/${{ github.repository_owner }}/argus:${{ github.ref_name }}
               ghcr.io/${{ github.repository_owner }}/argus:latest
             build-args: |
               BUILD_VERSION=${{ github.ref_name }}
               BUILD_DATE=${{ github.run_id }}
               BUILD_REF=${{ github.sha }}
           timeout-minutes: 120
         ```
         Notes (RESEARCH Pitfall 4): omit BUILD_ARCH from build-args — a single value cannot be per-arch in a unified build and the Supervisor routes on manifest platform entries, not the io.hass.arch label. Do NOT add `--prefer-binary` to build-args; it already lives in the Dockerfile pip step — just do not override it.
      8. Expose the resolved tag for the downstream job via a step output:
         ```yaml
         - name: Export image ref
           id: imgref
           run: echo "ref=ghcr.io/${{ github.repository_owner }}/argus:${{ github.ref_name }}" >> "$GITHUB_OUTPUT"
         ```
      Add a job-level `outputs:` mapping `image_ref: ${{ steps.imgref.outputs.ref }}` so the image-facts job consumes the exact tag.

    For `workflow_dispatch` runs (no tag) `github.ref_name` resolves to the branch name; that is acceptable — the gates still run against whatever tag was pushed. Do not add tag-vs-branch branching logic; keep the workflow simple.

  <files>.github/workflows/build.yml</files>

  <action>
    Add a second job `image-facts` to `.github/workflows/build.yml` with `needs: build-and-push`, `runs-on: ubuntu-latest`, and `timeout-minutes: 30`. Read the image ref from `${{ needs.build-and-push.outputs.image_ref }}` into an env var `IMG` for all steps.

    Steps:
      1. `docker/setup-qemu-action@v3` (needed to `docker run --platform linux/arm64` under emulation for the torch smoke test).
      2. `docker/login-action@v3` against ghcr.io (same as Task 1) — needed to inspect/pull the just-published private package.
      3. Manifest arch gate (RESEARCH Pattern 3 step 1):
         ```bash
         docker buildx imagetools inspect --raw "$IMG" \
           | jq -e '(.manifests // error("not a manifest list"))
                    | map(.platform.architecture)
                    | (contains(["amd64"]) and contains(["arm64"])) or error("missing arch")'
         ```
      4. torch-free gate per arch (RESEARCH Pattern 3 step 2 — assert import FAILS):
         ```bash
         for plat in linux/amd64 linux/arm64; do
           if docker run --rm --platform "$plat" "$IMG" python3 -c "import torch" 2>/dev/null; then
             echo "FAIL: torch importable on $plat"; exit 1
           fi
           echo "OK torch-free on $plat"
         done
         ```
      5. Size gate < 2 GB compressed per arch (RESEARCH Pattern 3 step 3 + Pitfall 5 — use compressed layer sizes from the per-arch image manifest, NOT `docker image ls`):
         For each arch digest in the index, inspect the per-arch manifest and sum `config.size + (layers[].size)`; assert the total < 2147483648. Implement as a bash loop using `docker buildx imagetools inspect --raw "$IMG"` to enumerate per-arch manifest digests, then `docker buildx imagetools inspect --raw "$IMG@<digest>"` to read each per-arch manifest's `layers[].size`, summing with jq and asserting `< 2147483648`. Fail the step (exit 1) if any arch exceeds the limit. Echo the measured MB per arch for the CI log.

    Each gate is its own named step so a failure is attributable. Use `bash` shell explicitly (`shell: bash`) on the script steps.
  </action>

  </verify>

YAML parses.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub-hosted runner → GHCR | CI authenticates and pushes a package consumed by end-user HA hosts |
| Release tag event → workflow | A tag push is the untrusted trigger that initiates a publish |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Elevation of Privilege | GITHUB_TOKEN scope | mitigate | `permissions:` block scopes to `contents: read` + `packages: write` only — no broader write. No PAT used. |
| T-04-02 | Tampering | Published image torch/size facts | mitigate | image-facts job re-inspects the *published* manifest (not a local build cache), asserting both arches + torch-free + size before the artifact is trusted. |
| T-04-03 | Spoofing | Action supply chain | accept | Actions referenced by major-version tag (v3/v4/v6) per RESEARCH; SHA-pinning is a documented hardening upgrade, deferred (low value for a single-maintainer personal add-on). |
| T-04-04 | Information Disclosure | GHCR package visibility | accept | First push defaults private; making it Public is a deliberate UAT step (RESEARCH Open Q2), not a CI auto-action. |
</threat_model>
