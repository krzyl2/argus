## Layout Shell

The layout shell (`wwwroot/templates/base.html` or a Minimal API `Results.Content` helper)
is the single reusable wrapper for all phase UIs. Phase 1 defines it; Phases 2–4 extend it.

```
┌─────────────────────────────────────────────────────────┐
│  <base href="{ingressPath}/">   (in <head>)             │
│                                                         │
│  <header class="argus-header">                          │
│    Argus  ·  [optional nav placeholder]                 │
│  </header>                                              │
│                                                         │
│  <main class="argus-main">                              │
│    {page content — placeholder card in Phase 1}         │
│  </main>                                                │
│                                                         │
│  <footer class="argus-footer">                          │
│    optional version / status strip                      │
│  </footer>                                              │
└─────────────────────────────────────────────────────────┘
```

Rules:
- `<base href="{ingressPath}/">` is emitted in `<head>` on every page response so that
  relative hrefs and static asset URLs resolve correctly through the Supervisor proxy.
- `X-Ingress-Path` middleware sets `context.Request.PathBase` per-request before `UseRouting`.
- Both mechanisms are required (PathBase for server-side redirect helpers / LinkGenerator;
  `<base>` for browser-relative asset resolution). See CONTEXT.md PathBase strategy.
- The layout uses `prefers-color-scheme` media query to adapt to HA light/dark themes.
- No JavaScript beyond htmx is loaded on the placeholder page in Phase 1.

## Typography

Four sizes, two weights. System font — no web font download (air-gapped, HA-native feel).

| Role | Size | Weight | Line Height | CSS Class |
|------|------|--------|-------------|-----------|
| Body | 14px | 400 | 1.5 | `argus-body` (default; applies to `body`) |
| Label | 12px | 600 | 1.4 | `argus-label` (form labels, metadata captions) |
| Heading | 20px | 600 | 1.2 | `argus-heading` (page-level and section headings) |
| Display | 28px | 600 | 1.15 | `argus-display` (hero text on placeholder page only) |

Notes:
- 14px body matches HA panel default; 20px/28px keep headings proportional without competing
  with HA's own chrome.
- Exactly 2 weights: 400 (regular) and 600 (semibold). No bold (700) — too heavy inside an iframe.

CSS custom properties:

```css
--font-family: system-ui, -apple-system, BlinkMacSystemFont, "Segoe UI", Roboto, sans-serif;
--font-size-body:    14px;
--font-size-label:   12px;
--font-size-heading: 20px;
--font-size-display: 28px;
--font-weight-regular:  400;
--font-weight-semibold: 600;
--line-height-body:    1.5;
--line-height-label:   1.4;
--line-height-heading: 1.2;
--line-height-display: 1.15;
```

## Color

The UI renders inside a Home Assistant Ingress iframe. Colors must feel native to HA — not
clash with it. Use CSS `prefers-color-scheme` to provide both light and dark palettes.

The color contract below defines the tokens; actual hex values adapt per scheme. Token names
are the source of truth for the executor; use the `--color-*` CSS custom properties.

| Role | Light Value | Dark Value | Usage |
|------|-------------|------------|-------|
| Dominant (60%) — surface | `#f8f9fb` | `#1c1c1e` | Page background, main content area |
| Secondary (30%) — element | `#ffffff` | `#2c2c2e` | Cards, header bar, footer strip |
| Accent (10%) — interactive | `#0d6efd` | `#3b9eff` | Reserved for specific elements only (see below) |
| Border | `#e0e0e0` | `#3a3a3c` | Card outlines, dividers |
| Text primary | `#111827` | `#f2f2f7` | Body text, headings |
| Text secondary | `#6b7280` | `#aeaeb2` | Captions, metadata, placeholder text |
| Destructive | `#dc3545` | `#ff453a` | Destructive actions only (Phase 3+) |
| Status ok | `#198754` | `#32d74b` | Health OK indicator (used in Phase 1 placeholder) |
| Status error | `#dc3545` | `#ff453a` | Health ERROR / service down indicator |

Accent (`--color-accent`) is reserved **exclusively** for:
1. The primary CTA button background (Phase 2+: "Save configuration")
2. Hyperlinks / text-links
3. Focus ring outline on interactive elements (2px solid, 2px offset)

Accent must NOT be applied to: headings, body text, card backgrounds, borders, status
indicators (those use `--color-status-ok` / `--color-status-error`).

CSS custom properties (light scheme shown; dark scheme in `@media (prefers-color-scheme: dark)`):

```css
:root {
  --color-surface:          #f8f9fb;
  --color-element:          #ffffff;
  --color-accent:           #0d6efd;
  --color-border:           #e0e0e0;
  --color-text-primary:     #111827;
  --color-text-secondary:   #6b7280;
  --color-destructive:      #dc3545;
  --color-status-ok:        #198754;
  --color-status-error:     #dc3545;
}

@media (prefers-color-scheme: dark) {
  :root {
    --color-surface:          #1c1c1e;
    --color-element:          #2c2c2e;
    --color-accent:           #3b9eff;
    --color-border:           #3a3a3c;
    --color-text-primary:     #f2f2f7;
    --color-text-secondary:   #aeaeb2;
    --color-destructive:      #ff453a;
    --color-status-ok:        #32d74b;
    --color-status-error:     #ff453a;
  }
}
```

## Placeholder Page (Phase 1 deliverable)

The Phase 1 page is the **only screen** this phase ships. It proves the Ingress plumbing
resolves end-to-end. No interactive elements beyond what is needed for proof.

### Content

```
[Header: "Argus"]

[Display heading: "Argus is running"]
[Body: "Configuration UI coming soon. Sensor anomaly detection is active."]

[Status row: green dot + "Detector connected" — or red dot + "Detector unreachable"]

[Footer: "v{version}" — populated server-side from assembly version]
```

### Proof criteria (maps to Phase 1 success criteria)

- `htmx.min.js` served via Ingress URL returns HTTP 200 (not 404).
- `argus.css` served via Ingress URL returns HTTP 200.
- The heading renders correctly inside HA's "Open Web UI" panel.
- The page adapts color scheme to HA light/dark theme.

### htmx conventions (foundation for Phase 2–4)

These conventions are established in Phase 1 and must be followed in all later phases:

| Convention | Rule |
|------------|------|
| htmx version | 2.0.10 — exact pinned version, committed to `wwwroot/js/htmx.min.js` |
| Script loading | `<script src="js/htmx.min.js" defer></script>` — defer, relative to `<base>` |
| Swap target | Always a named `<div id="...">` inside `<main>` — never `body` or `<main>` directly |
| Error handling | `hx-on:htmx:responseError` on `<body>`: swap error banner into `#argus-flash` div |
| Request indicator | `hx-indicator="#argus-spinner"` on all mutating requests (Phase 2+) |
| Push URL | `hx-push-url="false"` — Ingress iframe; no browser history manipulation |
| CSRF | Not required — single-user, Ingress-authenticated, no external origin access |

## Accessibility Baseline

Minimum requirements for Phase 1 (foundation for all later phases):

- `<html lang="en">` on every page response.
- All interactive elements (links, buttons) have visible focus ring using `--color-accent` at 2px solid, 2px offset.
- Color contrast: `--color-text-primary` on `--color-surface` meets WCAG AA (4.5:1 minimum) in both light and dark schemes.
- Status indicators use both color and a text label (never color alone) — e.g., dot + "Detector connected".
- No ARIA attributes required on the Phase 1 placeholder (no interactive widgets). ARIA contract defined in Phase 2 UI-SPEC when the entity picker is introduced.
