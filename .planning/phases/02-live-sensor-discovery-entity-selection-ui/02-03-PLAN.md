<objective>
Build the entity-picker UI and its three endpoints on top of the Wave-1 registry + glob resolver:
a full page (`GET /sensors`), an htmx search fragment (`GET /api/sensors`), and the save endpoint
(`POST /api/sensors/save`) that expands patterns, writes the resolved `entities.yaml` via the Phase-1
`ConfigWriter`, persists raw patterns under a `_patterns:` key, and creates the `/data/.ui_config_present`
lock file. Add the Phase-2 CSS classes to `argus.css` using existing tokens only.

Purpose: This is the user-facing deliverable for UI-02 (list + search + tracked distinction + selection)
and completes CFG-02 (patterns honored + persistence + guard activated by the lock file). The lock-file
guard READ was landed in Wave 1 (plan 02-02); this plan lands the WRITE — so a UI save + restart now
preserves config end-to-end.
Output: EntityPickerPage, argus.css Phase-2 blocks, three endpoints in Program.cs, save LogEvents,
and tests.

Assumptions (make explicit for the verifier):
- Phase 2 has NO live hot-reload. A save writes entities.yaml + the lock file; the running pipeline
  reflects the new selection on the NEXT add-on restart ("next pipeline cycle" per UI subheading).
  Do NOT add ILiveEntitiesConfig, inner-CTS restart, or MQTT retraction — that is Phase 3.
- Detector assignment UI + per-entity params are Phase 3; new entities get `hst` with empty params.
- Full validate_session middleware is Phase 4; Phase 2 uses interim IP/header acceptance.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser (via Supervisor Ingress proxy) → GET/POST endpoints | Untrusted form input: entity_id list + include/exclude glob patterns |
| Save handler → /data/entities.yaml + /data/.ui_config_present | Writes to the trusted add-on volume; must be atomic and safely serialized |
| Server-rendered HTML → browser | entity_id / friendly_name / patterns echoed back into the page (stored-XSS surface) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-07 | Tampering | Stored XSS via entity_id/friendly_name/patterns rendered into the picker page | mitigate | `WebUtility.HtmlEncode` EVERY user-originated string in EntityPickerPage (checkbox value=, display text, textarea contents, no-results query). EntityPickerPageTests asserts a `<script>` entity_id is encoded |
| T-02-08 | Tampering | Unsafe YAML construction of entities.yaml from untrusted entity_ids | mitigate | Serialize the entity list via YamlDotNet SerializerBuilder (never string-format entity ids into YAML). Only the fixed `_patterns:` block is a literal string, and its list items are patterns the operator typed — serialize/quote them via YamlDotNet-consistent formatting; write through ConfigWriter (atomic temp+rename+SemaphoreSlim) |
| T-02-09 | Spoofing / EoP | Unauthenticated access to config-write endpoint | mitigate | Interim auth: accept only requests carrying `X-Ingress-Path` OR from `172.30.32.2`/loopback via `RemoteIpAddress` (real TCP peer, not spoofable X-Forwarded-For). No `ports:` exposed, so all external traffic transits the Supervisor. Full validate_session deferred to Phase 4 — documented ACCEPTED interim risk |
| T-02-10 | Denial of Service | Concurrent save + config file-watch leaving a partial write | mitigate | ConfigWriter uses temp-then-`File.Move(overwrite)` + `SemaphoreSlim(1)` (Phase 1) — no partial reads |
| T-02-11 | Information Disclosure | Save-error banner leaking internal exception detail | mitigate | Error banner returns a generic HtmlEncoded reason ("unexpected error"/"disk error"); full exception goes to the add-on log only (UiSaveFailed) |
</threat_model>
