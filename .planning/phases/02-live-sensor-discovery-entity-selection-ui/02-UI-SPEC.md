# Phase 2 — UI Design Contract

> Visual and interaction contract for the entity picker screen: live sensor discovery,
> searchable flat list, tracked-pill distinction, include/exclude pattern filters,
> and explicit Save with success/error banner feedback.
>
> This phase EXTENDS the Phase 1 design foundation — no tokens, scales, or colors are
> redefined here. Every `--color-*`, `--space-*`, `--font-*` token reference below
> resolves to the values already declared in `wwwroot/css/argus.css`. Only NEW
> component classes needed for the picker screen are specified.
>
> Sources: 01-UI-SPEC.md (design foundation), 02-CONTEXT.md (UX decisions), REQUIREMENTS.md
> (UI-02, CFG-02). Generated by gsd-ui-researcher.

| Role | Light Value | Dark Value | Usage |

Accent (`--color-accent`) is reserved **exclusively** for (same contract as Phase 1, plus):
1. The primary CTA button background ("Save configuration")

4. Hyperlinks / text-links

surface-tinted overlay), heading text, filter labels.

New semantic token for the "tracked" pill — mapped to existing tokens, not a new variable:
- Tracked pill background: `--color-status-ok` (green)
- Tracked pill text: `#ffffff` (white, both schemes — sufficient contrast against green)

(`.argus-header` / `.argus-main` / `.argus-footer`) with `.argus-main` max-width extended

- `#argus-sensor-list` is the htmx swap target for list content (search filter, htmx poll refresh).

  save bar adjacent to the button. Hidden by default (`display:none`); htmx activates via `.htmx-request`.
- The filter panel uses a simple two-column CSS grid (50%/50%) on screens ≥ 600px; stacks single
  column below 600px.
- `max-width: 880px` is set inline on `.argus-main` override; the CSS class default (720px) is NOT
  changed — Phase 3 and 4 may need the wider canvas too, but the default is preserved for backward
  compatibility with the placeholder.

JavaScript timer complexity.

## htmx Interaction Contract

All Phase 1 htmx conventions apply unchanged. Phase 2 additions:

| Interaction | Element | htmx attributes | Server response |
|-------------|---------|-----------------|-----------------|
| Live search | `<input class="argus-search__input" type="search">` | `hx-get="/api/sensors" hx-target="#argus-sensor-list" hx-trigger="keyup changed delay:200ms" hx-include="[name='q']" hx-push-url="false"` | HTML fragment: new list rows for `#argus-sensor-list` |
| Save selection | `<form id="argus-picker-form">` | `hx-post="/api/sensors/save" hx-target="#argus-flash" hx-swap="innerHTML" hx-indicator="#argus-spinner" hx-push-url="false"` on the form | HTML fragment: `.argus-banner--success` or `.argus-banner--error` swapped into `#argus-flash` |
| Error handling | `<body>` | `hx-on:htmx:responseError="document.getElementById('argus-flash').innerHTML='...'"`  | Banner fragment (same `.argus-banner--error` structure) |

Form submission: the entire `<form id="argus-picker-form">` wraps the list and filter panel.
Checkboxes use `name="entities"` (multi-value); filter textareas use `name="include_patterns"`
and `name="exclude_patterns"`. A single POST body carries all values.

The Save button sits inside the form (type="submit"); htmx intercepts it via `hx-post` on the
form element. The button gains `.htmx-request` during flight (disabling it via CSS).

## Screen States

### Loading state (initial page load)

The full page is server-rendered on GET. No skeleton loader needed — the sensor list is
populated synchronously from `IHaSensorRegistry` (in-memory snapshot). Page load latency
is negligible (local LAN, single-user).

### Empty state — no sensors discovered (HA not yet connected)

Displayed inside `#argus-sensor-list` when `IHaSensorRegistry` returns an empty list.

```
<div class="argus-empty">
  <p class="argus-body">No sensors found.</p>
  <p class="argus-label">Argus has not yet received a sensor snapshot from Home Assistant.
  Check that the add-on can reach the Supervisor proxy and that the detector is running.</p>
</div>
```

CSS:
```
.argus-empty             — padding: var(--space-lg) var(--space-md); text-align: center;
                           display: flex; flex-direction: column; gap: var(--space-xs);
                           color: var(--color-text-secondary)
```

### Search no-results state

When the search query matches zero sensors, render the same `.argus-empty` block inside the
list with different copy (see Copywriting).

### Save success state

Server returns a `.argus-banner--success` fragment swapped into `#argus-flash`:

```html
<div class="argus-banner argus-banner--success"
     role="status" aria-live="polite">
  Configuration saved. {N} entities tracked.
</div>
```

The `role="status"` and `aria-live="polite"` announce the result to screen readers without
interrupting focus.

### Save error state

Server returns a `.argus-banner--error` fragment swapped into `#argus-flash`:

```html
<div class="argus-banner argus-banner--error"
     role="alert" aria-live="assertive">
  Save failed. {reason}. Check the add-on log for details.
</div>
```

`role="alert"` + `aria-live="assertive"` for error announcement (more urgent than success).

## API Endpoints (UI contract only — implementation detail in plan)

These endpoints are the contract between the picker page and the server. Implementation is
determined by the planner; this section defines the contract the UI depends on.

| Method | Path | Request | Response |
|--------|------|---------|----------|
| GET | `/sensors` | query param `?q=` (optional search string) | Full picker page HTML (on initial load) or `#argus-sensor-list` fragment (on search) |
| GET | `/api/sensors` | query param `?q=` | `#argus-sensor-list` HTML fragment (list rows only — htmx target) |
| POST | `/api/sensors/save` | form-encoded: `entities[]=...`, `include_patterns`, `exclude_patterns` | `.argus-banner--success` or `.argus-banner--error` HTML fragment |

The GET `/sensors` full-page response includes the complete layout shell (header, main, footer)
with `<base href>` and script tags. The GET `/api/sensors` fragment response is a bare list —
no shell. The POST response is a bare banner fragment — no shell.

## Accessibility Contract

Extends Phase 1 baseline. Additional requirements for Phase 2:

- Every `<input type="checkbox">` is wrapped in a `<label>` that covers the full row — the
  entire row is the click/tap target. This satisfies the 44×44px minimum without a custom
  component.
- Search `<input>` has `aria-label="Filter entities"` (no visible label; the placeholder is
  not a substitute for an accessible name).
- The sensor list is a `<form>` with `<ul>` (list semantics) inside. Each row is a `<li>`.
  This gives screen readers "X items" count.
- Save button has no `aria-label` needed — its text label "Save configuration" is descriptive.
- `#argus-spinner` has `aria-hidden="true"` — it is a purely visual indicator; the save result
  banner (`role="status"` or `role="alert"`) is the accessible feedback channel.
- Both banner variants use `role="status"` (success) and `role="alert"` (error) as specified
  in Screen States above.
- Color is never the sole differentiator: the "tracked" pill carries the text "tracked" in
  addition to its green background.
- Keyboard navigation: Tab reaches checkbox → next checkbox → pattern textareas → Save button.
  No custom focus management needed (native form element order is correct by DOM order).
