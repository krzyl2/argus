<objective>
Migrate every EntitiesConfig consumer to the live `ILiveEntitiesConfig` singleton and turn `HaListenerWorker`
into an inner-CTS restart loop that reloads the streaming pipeline on `ConfigChanged`, retracts removed
entities, and republishes discovery for added ones — the CFG-04 reload mechanism. All constructor signature
changes, the Program.cs DI registration swap, and the consumer rewiring land together so the solution
compiles as one atomic change.

Purpose: Without this migration a swap has no effect — the workers keep reading their ctor-captured stale
config. This is the most invasive cross-cutting change of the phase (per CONTEXT.md).
Output: rewired DI + three migrated consumers + a restart loop that keeps MQTT/gRPC alive and reloads HST
streaming in < 2 s (SC2) while retracting removed entities within ≤30 s (SC4).
</objective>

<halistener_loop>
<!-- Inner-CTS restart loop skeleton — 03-RESEARCH.md Q2 + Pitfalls 1/3. Encode these correctness properties: -->
1. Subscribe ConfigChanged once before the loop; unsubscribe in finally.
2. Handler: capture `var cts = innerCts; if (cts is not null && !cts.IsCancellationRequested) cts.Cancel();` (null-safe; Pitfall 1).
3. Each iteration: read `_liveConfig.Get()`; if `lastConfig != null` run the retraction diff + discovery re-publish for added entities BEFORE creating the new inner CTS; set `lastConfig = current`.
4. `innerCts = CancellationTokenSource.CreateLinkedTokenSource(stoppingToken)`; run `_scoreStreamPipeline.RunAsync(_haEventSource.ReadAllAsync(innerCts.Token), innerCts.Token)`.
5. `catch (OperationCanceledException) when (innerCts.IsCancellationRequested && !stoppingToken.IsCancellationRequested)` → log ConfigReloadTriggered, re-loop. An OCE while stoppingToken is cancelled is NOT caught → propagates as clean host shutdown.
6. finally: `var toDispose = innerCts; innerCts = null; toDispose?.Dispose();` — null BEFORE Dispose so the handler never touches a disposed CTS (Pitfall 3).
Retraction diff: new id set = HashSet(newConfig.Entities.EntityId, OrdinalIgnoreCase); removed = oldConfig.Entities where id not in new → `DiscoveryPublisher.RetractAsync(_mqtt, removed, stoppingToken)`; log MqttRetractionPublished. Added entities (in new, not in old) → `DiscoveryPublisher.PublishAllAsync(_mqtt, added, stoppingToken)` (idempotent — research Q8/Pitfall 4). Use stoppingToken (not innerCts) for retraction/publish so the broker call survives the pipeline cancel.
</halistener_loop>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| save-endpoint thread → worker threads | ConfigChanged fires on the request thread; workers read Get() on their own threads (shared mutable reference) |
| reload trigger → running pipeline/host | A swap forces a pipeline restart — must not tear down the host, MQTT, or gRPC, and must not be abusable for DoS |
| orchestrator → MQTT broker | Retraction/republish on reload must target only changed entities |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-05 | Denial of Service | HaListenerWorker restart loop | mitigate | Inner CTS is linked to stoppingToken; only the inner CTS is cancelled on reload — MQTT + gRPC singletons stay alive (never torn down). OCE guard requires `!stoppingToken.IsCancellationRequested` so host shutdown always propagates and exits the loop. |
| T-03-06 | Denial of Service | Reload trigger abuse (rapid saves) | accept | Reload is reachable only behind the Phase-2 interim auth (Supervisor IP 172.30.32.2 / loopback); a rapid re-save at worst collapses to one missed reload trigger (research Pitfall 1) — no host crash. Full validate_session is Phase 4. Single-user LAN tool. |
| T-03-07 | Tampering | Retraction/republish diff | mitigate | Diff computed from server-controlled EntityConfig id sets (OrdinalIgnoreCase); RetractAsync only touches removed-entity topics; PublishAllAsync republish is idempotent (retained, same payload). |
| T-03-08 | Denial of Service | Inner CTS disposal race with handler | mitigate | null-before-Dispose pattern (Pitfall 3): `innerCts = null` set before `Dispose()`; handler null-checks — no ObjectDisposedException. Tested under rapid ConfigChanged. |
| T-03-09 | Tampering | Torn read of swapped config mid-restart | mitigate | ScoreStreamPipeline reads `_liveConfig.Get()` once at RunAsync entry (post-swap, single volatile read) — no partial config observed. |
</threat_model>
