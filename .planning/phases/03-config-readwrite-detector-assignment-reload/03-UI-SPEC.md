# Phase 3 — UI Design Contract

> Visual and interaction contract for the detector-assignment screen: per-entity expandable
> disclosure rows, detector-type dropdown (HST/MAD/STL), multi-detector add/remove, typed
> parameter fields with defaults, and the "Saved — pipeline reloading…" → success banner.
>
> This phase EXTENDS the Phase 2 entity picker. No tokens, scales, or colors are redefined.
> Every `--color-*`, `--space-*`, `--font-*` token reference below resolves to values already
> declared in `wwwroot/css/argus.css`. All Phase 1 and Phase 2 classes are inherited unchanged.
> Only NEW component classes and interaction states required for the detector-assignment section
> are specified here.
>
> Sources: 01-UI-SPEC.md (design foundation), 02-UI-SPEC.md (picker screen inherited),
> 03-CONTEXT.md (all UX decisions locked), REQUIREMENTS.md (UI-03, CFG-03, CFG-04),
> EntitiesConfig.cs (HstParams defaults). Generated by gsd-ui-researcher.

## Disclosure Toggle — Interaction Contract

The disclosure toggle exposes/hides per-entity detector configuration without a server round-trip.

**Implementation approach:** Use the HTML `<details>`/`<summary>` element pair. The browser
provides native expand/collapse semantics, keyboard support, and correct ARIA without custom JS.

```html
<details class="argus-detectors-details">
  <summary class="argus-disclosure-toggle">
    Detectors (2)
  </summary>
  <div class="argus-detectors-panel">
    <!-- detector entries -->
  </div>
</details>
```

Rules:
- `<details>` is placed immediately below the checkbox `<label>` inside each tracked `<li>`.
- The `open` attribute is NOT set by default — all panels start collapsed.
- `<summary>` text: `Detectors ({N})` where `{N}` is the current detector count for that entity
  (server-substituted). When 0 detectors: `Detectors (none)`.
- The CSS disclosure triangle on `<summary>` is the browser's native marker (no custom SVG
  needed). Override with `list-style: none` and inject a CSS `▶`/`▼` via `::before` pseudo
  to match the design system font (see New Component Classes).
- Touch/click anywhere on the `<summary>` expands/collapses the panel.
- No htmx attributes on `<details>` or `<summary>` — this is pure browser behavior.
- Keyboard: `Space` / `Enter` on focused `<summary>` toggles; this is native `<details>` behavior.

## Detector Entry — Layout and Fields

Each detector assigned to an entity is rendered as a `.argus-detector-entry` block inside the
disclosure panel. Entries are ordered by insertion order (reflect YAML order when pre-filling).

### Detector type dropdown

```html
<select class="argus-detector-select"
        name="detectors[{entity_idx}][{det_idx}][name]"
        aria-label="Detector type for {entity_id}">
  <option value="hst">HST</option>
  <option value="mad">MAD</option>
  <option value="stl">STL</option>
</select>
<span class="argus-timing-caption" data-detector-name="{current_name}">
  {timing_label}
</span>
```

- The selected `<option>` reflects the current `DetectorConfig.Name` from entities.yaml (pre-filled).
- `{timing_label}` is server-rendered: `streaming (live, ~2 s reload)` for HST; `batch (runs every N min)` for MAD and STL.
- The timing caption is static on initial render. It does NOT update dynamically on dropdown
  change in Phase 3 (dynamic update is Phase 4 enhancement — client-side JS not in scope).
- Name field uses `name="detectors[{entity_idx}][{det_idx}][name]"` indexed form encoding
  (e.g. `detectors[0][0][name]=hst`). The server reconstructs per-entity detector lists from
  the indexed POST body.

### Parameter grid

Each detector type shows its own parameter set. The server renders only the fields relevant
to the currently assigned type (the full field set is pre-known and type-specific).

**HST parameters** (7 fields — 2-column grid):

| Field name (HTML `name=`) | Label | Default | Input type |
|---------------------------|-------|---------|------------|
| `detectors[i][j][params][window]` | `window` | `250` | `number` |
| `detectors[i][j][params][n_trees]` | `n_trees` | `25` | `number` |
| `detectors[i][j][params][high_threshold]` | `high_threshold` | `0.7` | `number` (step="0.01") |
| `detectors[i][j][params][low_threshold]` | `low_threshold` | `0.3` | `number` (step="0.01") |
| `detectors[i][j][params][min_consecutive]` | `min_consecutive` | `3` | `number` |
| `detectors[i][j][params][frozen_window]` | `frozen_window` | `10` | `number` |
| `detectors[i][j][params][frozen_variance_threshold]` | `frozen_variance` | `0.001` | `number` (step="0.0001") |

Defaults are sourced from `HstParams` in `EntitiesConfig.cs` (D-09/D-11/D-12 values).
`frozen_variance_threshold` label is truncated to `frozen_variance` to fit the 2-column grid;
the full param name is preserved in the `name` attribute.

**MAD parameters** (2 fields — 2-column grid):

| Field name | Label | Default | Input type |
|------------|-------|---------|------------|
| `detectors[i][j][params][threshold]` | `threshold` | `3.5` | `number` (step="0.1") |
| `detectors[i][j][params][window]` | `window` | `20` | `number` |

**STL parameters** (3 fields — 2-column grid, last field spans both columns):

| Field name | Label | Default | Input type |
|------------|-------|---------|------------|
| `detectors[i][j][params][period]` | `period` | `24` | `number` |
| `detectors[i][j][params][seasonal]` | `seasonal` | `7` | `number` |
| `detectors[i][j][params][threshold]` | `threshold` | `3.0` | `number` (step="0.1") |

Pre-fill rule: if the key is present in entities.yaml `Params`, use that value; if absent,
render the default value above as the `<input value="...">` attribute.

### Remove detector button

```html
<button type="button"
        class="argus-btn argus-btn--destructive-ghost"
        aria-label="Remove this detector from {entity_id}">
  Remove detector
</button>
```

- `type="button"` — not a submit button; must NOT trigger form POST.
- Clicking removes the `.argus-detector-entry` from the DOM. This is the one Phase 3 client-side
  interaction that requires a small inline script (no htmx, no library). Implementation: 3-line
  `onclick` that calls `this.closest('.argus-detector-entry').remove()`.
- No confirmation dialog for Remove — the detector entry is only permanently removed on Save.
  The user can abandon changes by navigating away.
- After removal, the `<details>` summary counter `Detectors ({N})` is NOT updated dynamically
  in Phase 3 (it reflects the server-rendered count; Phase 4 may add JS to live-update it).
- `aria-label` names the entity being affected.

### Add detector button

```html
<button type="button"
        class="argus-btn argus-btn--add-detector"
        data-entity-id="{entity_id}"
        hx-get="/api/detectors/new-entry?entity_idx={entity_idx}&det_idx={next_det_idx}"
        hx-target=".argus-add-detector-row[data-entity-idx='{entity_idx}']"
        hx-swap="beforebegin"
        hx-push-url="false">
  + Add detector
</button>
```

- htmx GET fetches a new empty `.argus-detector-entry` HTML fragment from the server and
  inserts it before the add-detector row. The server returns a pre-rendered entry with HST
  defaults (HST is the default type for new entries).
- `{next_det_idx}` is the current count of existing entries for that entity (server knows it;
  passes it as a data attribute on the button).
- The button is wrapped in a `<div class="argus-add-detector-row" data-entity-idx="{i}">` which
  acts as both the hx-target sentinel and a layout container.
- `hx-indicator="#argus-spinner"` on the button (same spinner as Save — there is only one spinner
  in the page; concurrent requests are not expected in a single-user tool).

|-------------|---------|-----------------|-----------------|
| Add detector | `<button class="argus-btn--add-detector">` | `hx-get="/api/detectors/new-entry?entity_idx={i}&det_idx={j}" hx-target=".argus-add-detector-row[data-entity-idx='{i}']" hx-swap="beforebegin" hx-indicator="#argus-spinner" hx-push-url="false"` | HTML fragment: one `.argus-detector-entry` block (HST, all defaults) |
| Save (extended) | `<form id="argus-picker-form">` | Inherited from Phase 2 — `hx-post="/api/sensors/save"` | HTML fragment: `.argus-banner--reloading` (transient) then `.argus-banner--success` (final) |
| Remove detector | `<button class="argus-btn--destructive-ghost">` | No htmx — inline `onclick="this.closest('.argus-detector-entry').remove()"` | No server request |

**Save form encoding:** The Phase 2 POST body (entity checkboxes + patterns) is extended with
indexed detector fields:

```
detectors[0][0][name]=hst
detectors[0][0][params][window]=250
detectors[0][0][params][n_trees]=25
detectors[0][0][params][high_threshold]=0.7
...
detectors[0][1][name]=mad
detectors[0][1][params][threshold]=3.5
detectors[0][1][params][window]=20
detectors[1][0][name]=stl
...
```

`{entity_idx}` is the 0-based index of entities as ordered in the submitted checkbox list.
The server matches entity index to entity_id by correlating with the `entities[]` checkbox
values in the same POST body.

## Screen States

### Collapsed tracked row

Default state. The tracked entity row shows the checkbox row + a collapsed `<details>` toggle.

```
[✓] sensor.living_room_temp   22.5 °C   [tracked]
[▶ Detectors (1)]
```

The disclosure toggle shows the current detector count as a parenthetical. No detector
details are visible in this state.

### Expanded tracked row — no detectors (first-time tracked entity)

When a newly-checked entity has no saved detectors, the panel opens empty with a default
HST entry pre-rendered (sane default per 03-CONTEXT.md decision: "default to hst with sane defaults").

```
[▼ Detectors (1)]
┌─────────────────────────────────────────┐
│ [HST ▼] streaming (live, ~2 s reload)   │
│ window [250]  n_trees [25]              │
│ high_threshold [0.7]  low_threshold[0.3]│
│ min_consecutive [3]                     │
│ frozen_window [10]  frozen_variance[0.001] │
│                              [Remove]   │
└─────────────────────────────────────────┘
[+ Add detector]
```

### Expanded tracked row — with pre-filled detectors

When entities.yaml contains existing detector assignments, each is pre-rendered as a
`.argus-detector-entry` with its saved param values (falling back to type defaults for
absent keys).

### Untracked row

An untracked entity row renders exactly as in Phase 2 — checkbox, entity_id, meta, no
disclosure section. When the user checks an untracked row, the form captures it but no
detector panel is shown until Save is processed and the page re-renders (the server
adds a default HST entry in the next full render). This simplification avoids the need for
a client-side htmx insertion on checkbox change in Phase 3.

### Reloading banner state (intermediate)

After a successful save POST, the server first returns this intermediate fragment into `#argus-flash`:

```html
<div class="argus-banner argus-banner--reloading"
     role="status" aria-live="polite">
  Saved — pipeline reloading…
</div>
```

Then the server completes the config swap + pipeline reload and returns the success fragment
(same POST response, or a follow-up htmx poll — implementation detail for the planner). The
intermediate state is visible during the reload gap (< 1 second for HST; up to the next batch
cycle for MAD/STL).

**Implementation note for planner:** The simplest Phase 3 approach is a single POST response
that returns the reloading banner. The pipeline reload happens synchronously in the request
handler (or is awaited within it), and the final success fragment is the one response. This
avoids the need for polling or SSE in Phase 3. The "reloading…" banner may flash briefly if
the reload is fast enough; that is acceptable.

### Save success state (final)

```html
<div class="argus-banner argus-banner--success"
     role="status" aria-live="polite">
  Saved — pipeline active. {N} entities tracked.
</div>
```

### Save error state

```html
<div class="argus-banner argus-banner--error"
     role="alert" aria-live="assertive">
  Save failed. {reason}. Check the add-on log for details.
</div>
```

### Empty detector panel

If an entity has all detectors removed via the Remove button (before Save), the disclosure
panel shows only the "+ Add detector" button:

```
[▼ Detectors (none — save will use HST defaults)]
[+ Add detector]
```

The "(none — save will use HST defaults)" parenthetical is a static copy hint; the server
will insert a default HST entry when saving an entity with an empty detector list.

## Accessibility Contract

Extends Phase 2 baseline.

- `<details>`/`<summary>` provides native keyboard expand/collapse and screen reader semantics
  without additional ARIA. Do NOT add `aria-expanded` manually — the browser manages it.
- The `<summary>` element is inherently focusable; Tab will reach each entity's disclosure
  toggle in DOM order.
- `<select class="argus-detector-select">` has `aria-label="Detector type for {entity_id}"`
  — the label identifies both the role and which entity it applies to (multiple selects on page).
- Param `<input>` elements are each wrapped in `.argus-param-field` with an explicit `<label
  for="{id}">` or use `aria-label`. The label text matches the field name (e.g. `window`,
  `n_trees`, `threshold`).
- Remove button: `aria-label="Remove this detector from {entity_id}"` names the affected entity.
- Add detector button: `aria-label="Add detector to {entity_id}"` names the affected entity.
- `#argus-spinner` retains `aria-hidden="true"` from Phase 2 — purely visual; banners carry
  accessible announcement.
- Reloading banner: `role="status"` + `aria-live="polite"`. Success: same. Error: `role="alert"`
  + `aria-live="assertive"`.
- Color alone does not distinguish detector types or validation states — text labels are always present.
- The Remove button uses both destructive color AND the text label "Remove detector" — never icon-only.
