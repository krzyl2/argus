# Phase 4 — UI Design Contract

> Visual and interaction contract for the validation layer: per-field error states on
> entity_id and detector-param inputs, a disabled Save button while any field is invalid,
> a validation summary banner returned by the server on rejection, and a warm-up disclosure
> note near the reload banner.
>
> This is the FINAL v3.0 phase and a SMALL UI delta. This phase EXTENDS Phases 1–3.
> No tokens, scales, colors, or existing component classes are redefined. Every
> `--color-*`, `--space-*`, `--font-*` token reference below resolves to values already
> declared in `wwwroot/css/argus.css`. All Phase 1, 2, and 3 BEM classes are inherited
> unchanged.
>
> Only the following are NEW in Phase 4:
> - `.argus-param-field--error` modifier (field highlight + inline message)
> - `.argus-btn--primary[disabled]` behavior spec (already partially in Phase 2 CSS; reinforced here)
> - `.argus-banner--validation` (server validation rejection banner)
> - `.argus-warmup-note` (warm-up disclosure near the reload/success banner)
> - Client-side inline JS validation (mirrors server rules; no library, no build step)
>
> Sources: 01-UI-SPEC.md (design foundation), 02-UI-SPEC.md (picker + banners),
> 03-UI-SPEC.md (param fields being validated), 04-CONTEXT.md (all decisions locked),
> REQUIREMENTS.md (UI-04, DOCS-02). Generated by gsd-ui-researcher.

## Validation Rules (design contract — mirrors server rules)

These rules define WHEN a field enters error state. They are locked in 04-CONTEXT.md
and must be mirrored exactly client-side by the inline JS.

### entity_id

- Rule: must match `^[a-z0-9_]+\.[a-z0-9_]+$`
- Trigger: on blur (field loses focus) AND on form submit attempt
- Error message: `Invalid entity ID. Use format domain.object_id (e.g. sensor.living_room_temp).`

### HST parameters

| Field | Rule | Error message |
|-------|------|---------------|
| `window` | integer ≥ 1 | `Must be a whole number ≥ 1.` |
| `n_trees` | integer ≥ 1 | `Must be a whole number ≥ 1.` |
| `high_threshold` | number in (0, 1], must be > `low_threshold` | `Must be between 0 and 1, and greater than low threshold.` |
| `low_threshold` | number in [0, 1), must be < `high_threshold` | `Must be between 0 and 1, and less than high threshold.` |
| `min_consecutive` | integer ≥ 1 | `Must be a whole number ≥ 1.` |
| `frozen_window` | integer ≥ 1 | `Must be a whole number ≥ 1.` |
| `frozen_variance_threshold` | number ≥ 0 | `Must be 0 or greater.` |

### MAD parameters

| Field | Rule | Error message |
|-------|------|---------------|
| `threshold` | number > 0 | `Must be greater than 0.` |
| `window` | integer ≥ 1 | `Must be a whole number ≥ 1.` |

### STL parameters

| Field | Rule | Error message |
|-------|------|---------------|
| `period` | integer ≥ 2 | `Must be a whole number ≥ 2.` |
| `threshold` | number > 0 | `Must be greater than 0.` |

### Detector name

- Rule: must be one of `hst`, `mad`, `stl` (case-insensitive; enforced by the `<select>` — unknown values cannot be submitted via normal form interaction)
- If received by server via a tampered POST: server rejects with validation banner: `Unknown detector type "{name}". Choose HST, MAD, or STL.`
- No client-side check needed (select options are the complete allowed set)

### Validation trigger timing

- **On blur** (field loses focus): validate the single field; show/clear its error state.
- **On input** (while a field is already in error state): re-validate immediately on each
  keystroke so the error clears as soon as the value becomes valid (fast feedback).
- **On form submit attempt**: validate ALL fields; prevent POST if any field is invalid.
  The Save button click is intercepted before htmx fires.

## New Component Classes (Phase 4 additions to argus.css)

### `.argus-param-field--error` modifier

Applied to `.argus-param-field` when its input fails validation. The modifier highlights
the input border and reveals the inline error message.

```
.argus-param-field--error .argus-param-field__input
                           — border-color: var(--color-status-error);
                             outline: none
.argus-param-field--error .argus-param-field__input:focus
                           — border-color: var(--color-status-error)
                             (overrides the accent focus border — error takes priority)
.argus-param-field--error .argus-param-field__error-msg
                           — display: block
                             (revealed by the modifier; hidden by default)
```

The modifier is toggled by the inline JS validation function. The server also returns
forms pre-decorated with this modifier when it rejects a save (via the htmx error fragment).

### `.argus-param-field__error-msg`

Inline error message element rendered inside `.argus-param-field`, immediately below the
`<input>`. Hidden by default; revealed only when `.argus-param-field--error` is present.

```
.argus-param-field__error-msg
                           — display: none;
                             font-size: var(--font-size-label);
                             font-weight: var(--font-weight-semibold);
                             line-height: var(--line-height-label);
                             color: var(--color-status-error);
                             margin-top: var(--space-xs)
```

HTML structure (added to every `.argus-param-field` in Phase 4):

```html
<div class="argus-param-field">
  <label class="argus-param-field__label" for="param-{entity_idx}-{det_idx}-{key}">
    {param_name}
  </label>
  <input class="argus-param-field__input"
         id="param-{entity_idx}-{det_idx}-{key}"
         type="number"
         name="detectors[{i}][{j}][params][{key}]"
         value="{value}"
         aria-describedby="param-{entity_idx}-{det_idx}-{key}-err"
         aria-invalid="false">
  <span class="argus-param-field__error-msg"
        id="param-{entity_idx}-{det_idx}-{key}-err"
        role="alert"
        aria-live="assertive">
    {error message text — server-rendered or JS-populated}
  </span>
</div>
```

When the JS validation sets a field invalid:
1. Add `argus-param-field--error` to the parent `.argus-param-field`.
2. Set `aria-invalid="true"` on the `<input>`.
3. Populate the `<span class="argus-param-field__error-msg">` with the error message string.

When a field becomes valid again:
1. Remove `argus-param-field--error` from the parent.
2. Set `aria-invalid="false"` on the `<input>`.
3. Clear the `<span>` text content (empty string — not `display:none` directly; the CSS
   modifier controls visibility).

### `.argus-banner--validation`

Server-side validation rejection banner. Swapped into `#argus-flash` when the server
rejects the POST (same target as the success/error banners from Phases 2–3).

```
.argus-banner--validation  — background-color: var(--color-status-error); color: #ffffff
                             (inherits `.argus-banner` padding, border-radius, typography)
```

This banner communicates that the server rejected the save due to validation errors. It
appears alongside (or instead of) the per-field error states when server and client disagree
(e.g. a client-side validation gap was bypassed). The server remains the authoritative check.

```html
<div class="argus-banner argus-banner--validation"
     role="alert" aria-live="assertive">
  Save blocked: {N} field(s) have invalid values. Correct the highlighted fields and try again.
</div>
```

If the server also returns per-field error decorations (the full picker fragment with
`.argus-param-field--error` pre-applied), the banner appears above the form as the
`#argus-flash` swap. Per-field errors are embedded in the returned fragment.

### `.argus-warmup-note`

Warm-up disclosure rendered near the reload/success banner — inside or directly below
`#argus-flash`. Appears on save success (pipeline reloaded) to set user expectation.

```
.argus-warmup-note         — font-size: var(--font-size-label);
                             font-weight: var(--font-weight-semibold);
                             line-height: var(--line-height-label);
                             color: var(--color-text-secondary);
                             margin-top: var(--space-xs)
```

It is NOT a separate banner — it is a `<p>` appended inside the success banner fragment,
visually subordinate to the main success message.

```html
<div class="argus-banner argus-banner--success"
     role="status" aria-live="polite">
  Saved — pipeline active. {N} entities tracked.
  <p class="argus-warmup-note">
    HST detectors need ~4 minutes of readings to warm up (window=250 at ~1 reading/s).
    Anomaly scores will be low until warm-up completes.
  </p>
</div>
```

The warm-up note is rendered only when the config contains at least one HST detector. If
the config contains only MAD/STL detectors, the note is omitted (batch detectors have no
comparable warm-up concept).

## Save Button — Disabled State

The Save button (`<button type="submit" class="argus-btn argus-btn--primary">`) is
DISABLED while any field in the form is invalid.

**CSS** (already declared in Phase 2, reinforced here for clarity):

```
.argus-btn[disabled],
.argus-btn.htmx-request   — opacity: 0.5; cursor: not-allowed; pointer-events: none
```

No new CSS rule is needed. The existing Phase 2 rule covers `[disabled]`. The inline JS
sets and clears the `disabled` attribute on the button as field validity changes.

**Logic** (inline JS):

- After each field validation run, count total invalid fields across the entire form.
- If count > 0: set `disabled` attribute on the Save button.
- If count === 0: remove `disabled` attribute from the Save button.

The button must NOT re-enable while htmx has a request in flight. The Phase 2 `.htmx-request`
CSS rule already covers the in-flight disabled state; the JS must not fight it.

## Inline JS Validation — Contract

The validation logic is a small inline `<script>` block at the bottom of the picker page,
rendered by `EntityPickerPage.cs`. It is NOT a separate `.js` file. It MUST NOT use any
external libraries, network calls, `eval`, or dynamic imports.

**Validation rules object** (authored in the script — source of truth for client-side):

```javascript
const PARAM_RULES = {
  // Key: matches the final segment of the input's name= attribute (the param key)
  // Applies to any input with matching param key regardless of entity/detector indices
  "window":                     { min: 1, integer: true },
  "n_trees":                    { min: 1, integer: true },
  "high_threshold":             { min: 0, max: 1, exclusive_min: true,
                                  cross: "gt:low_threshold" },
  "low_threshold":              { min: 0, max: 1, exclusive_max: true,
                                  cross: "lt:high_threshold" },
  "min_consecutive":            { min: 1, integer: true },
  "frozen_window":              { min: 1, integer: true },
  "frozen_variance_threshold":  { min: 0 },
  "threshold":                  { min: 0, exclusive_min: true },
  "period":                     { min: 2, integer: true }
};
```

**Event bindings** (all wired once via event delegation on the form element):
- `focusout` → validate the single field that lost focus
- `input` → re-validate the single field IF it currently has `argus-param-field--error`
- `submit` → validate all fields; call `event.preventDefault()` if any invalid; do NOT
  manually trigger htmx — htmx handles the POST if the native submit proceeds

No `DOMContentLoaded` wrapper is needed — the script tag is at the bottom of `<body>`
(before `</body>`), after the form elements are in the DOM.

**Inline JS size budget:** the entire script (including the rules object) MUST stay under
2 KB minified. This is enforceable by inspection at authoring time given the bounded rule set.

## htmx Interaction Contract

All Phase 1, 2, and 3 htmx conventions apply unchanged. Phase 4 modifications:

| Interaction | Change | Detail |
|-------------|--------|--------|
| Save POST (validation pass) | Unchanged | htmx fires normally; response is success/reloading banner |
| Save POST (client validation fail) | New: blocked | JS calls `event.preventDefault()` on `submit`; htmx never fires; per-field errors shown |
| Save POST (server validation fail) | New: banner | Server returns `.argus-banner--validation` fragment + optionally re-rendered form fragment into `#argus-flash` |
| Server-returned per-field errors | New: fragment | Server may return a complete form-fragment with `.argus-param-field--error` pre-applied on the relevant fields, swapped via `hx-swap="outerHTML"` targeting `#argus-picker-form` |

**Server response options on validation failure:**

Option A (banner only — simpler): return only the `.argus-banner--validation` fragment
into `#argus-flash`. Per-field errors not decorated server-side. Appropriate when the
server detects a single structural problem (e.g. unknown detector name from a tampered
POST body).

Option B (banner + form fragment): return a multi-fragment htmx OOB swap response:
- Swap `.argus-banner--validation` into `#argus-flash`
- Swap a re-rendered `#argus-picker-form` with `.argus-param-field--error` pre-applied

**Decision for planner:** Option A is sufficient for Phase 4. Option B is an enhancement
if server-side catches a case client-side missed. Use `hx-swap-oob` for the form re-render
if Option B is implemented. The planner decides; either option is compatible with this spec.

## Screen States

### Valid form state (baseline)

All fields pass their rules. The Save button is enabled (`no disabled attribute`).
No `.argus-param-field--error` modifiers present. No validation banner in `#argus-flash`.

### Invalid form state — one or more fields

One or more `.argus-param-field` elements carry the `--error` modifier:
- The input has `border-color: var(--color-status-error)` (1px solid, same border-radius as valid)
- The `.argus-param-field__error-msg` is visible with error text in `--color-status-error`
- The Save button carries `disabled` attribute → `opacity: 0.5`, `cursor: not-allowed`

```
[window] [  -5  ]
          Must be a whole number ≥ 1.    ← argus-param-field__error-msg (12px/600, status-error)

                  [ Save configuration ]  ← opacity: 0.5, pointer-events: none
```

The rest of the form remains fully interactive. The user can still edit, scroll, expand
disclosure panels, and add/remove detector entries. Only the Save is blocked.

### Server rejection state

The server rejects the POST. `#argus-flash` receives the validation banner:

```
┌──────────────────────────────────────────────────────────────────┐
│  Save blocked: 2 field(s) have invalid values.                   │
│  Correct the highlighted fields and try again.                   │
│                        [argus-banner--validation, role="alert"]  │
└──────────────────────────────────────────────────────────────────┘
```

If the server also returns per-field error decoration (Option B), the form re-renders with
highlighted fields. If not (Option A), the user sees only the banner and must identify the
fields themselves (acceptable for a single-developer tool where the client-side guard
should have caught it first).

### Save success state with warm-up note (HST present)

```
┌──────────────────────────────────────────────────────────────────┐
│  Saved — pipeline active. 3 entities tracked.                    │
│  HST detectors need ~4 minutes of readings to warm up            │
│  (window=250 at ~1 reading/s). Anomaly scores will be low until  │
│  warm-up completes.                                              │
│                        [argus-banner--success, role="status"]    │
└──────────────────────────────────────────────────────────────────┘
```

### Save success state without warm-up note (MAD/STL only)

```
┌──────────────────────────────────────────────────────────────────┐
│  Saved — pipeline active. 2 entities tracked.                    │
│                        [argus-banner--success, role="status"]    │
└──────────────────────────────────────────────────────────────────┘
```

## Accessibility Contract

Extends Phase 3 baseline.

- Every param `<input>` gains `aria-describedby="param-{entity_idx}-{det_idx}-{key}-err"`
  linking it to its error message `<span>`. This is set in the server-rendered HTML on all
  param fields, valid or invalid.
- The JS sets `aria-invalid="true"` when a field is invalid, `aria-invalid="false"` when
  valid. The initial server-rendered value is `aria-invalid="false"`.
- The `.argus-param-field__error-msg` span carries `role="alert"` and `aria-live="assertive"`.
  When JS populates the span text (from empty to a message), screen readers announce it.
  When JS clears the span (from message to empty), no announcement fires — intentional
  (clearing an error is not urgent information).
- The validation banner (`#argus-flash` swap) uses `role="alert"` + `aria-live="assertive"`
  (same as the Phase 2/3 error banner contract).
- The warm-up note inside the success banner inherits the parent's `role="status"` and
  `aria-live="polite"`. It is a `<p>` element with no additional ARIA — standard block text.
- The disabled Save button (`disabled` attribute present): screen readers announce it as
  unavailable. Do NOT use `aria-disabled` instead of the HTML `disabled` attribute —
  the native attribute correctly prevents form submission AND keyboard interaction.
- Keyboard flow is unchanged from Phase 3. Validation does not reorder focus. If a field
  fails on blur, focus stays where it naturally went (the next element). Error messages are
  announced by `aria-live` without stealing focus.
- Color alone does not indicate validity — the invalid state combines: red border color
  (visual), `aria-invalid="true"` (programmatic), AND visible error message text (textual).
