  <files>argus/rootfs/usr/local/bin/wait-detector.py, detector/tests/test_wait_detector.py</files>
  <action>
Create `wait-detector.py` as a synchronous gRPC health poller (use `import grpc` and `grpc_health.v1`, NOT `grpc.aio` — async is unnecessary complexity per ARCHITECTURE.md anti-patterns). Structure it for testability: expose a pure predicate `check_serving(addr, service, timeout=2)` that opens an insecure channel, calls `HealthStub.Check` with the given service name, and returns True only when the response status is SERVING, returning False on any exception or non-SERVING status. Expose a `wait_until_serving(addr, service, interval=1.0, max_attempts=None)` loop that calls `check_serving` until it succeeds, sleeping `interval` between attempts, and a `main(argv)` that reads `addr` (default `127.0.0.1:50051`) and `service` (default `argus.v1.DetectorService`) from argv and exits 0 once SERVING. Guard the entry point with `if __name__ == "__main__"` so the module is importable without running the loop. The default service name MUST match the name the detector registers (`argus.v1.DetectorService`, per `server.py` health set).

Create `detector/tests/test_wait_detector.py` following the loopback pattern in `test_local_mode.py`: start a `create_server(port=free_port, tls=False)` bound to 127.0.0.1, assert `check_serving("127.0.0.1:{port}", "argus.v1.DetectorService")` returns True; assert it returns False against an unused/closed port (use a bound-then-closed free port). Add `sys.path` handling so the test can import the script from `argus/rootfs/usr/local/bin/` (e.g. `importlib.util.spec_from_file_location`), or add the directory to `sys.path` in the test.

<task type="auto">
  <name>Task 2: s6 run + finish scripts for detector and orchestrator</name>
  <files>argus/rootfs/etc/services.d/detector/run, argus/rootfs/etc/services.d/detector/finish, argus/rootfs/etc/services.d/orchestrator/run, argus/rootfs/etc/services.d/orchestrator/finish</files>
  <action>
Create the four s6-overlay legacy-layout scripts (HA base image uses `cont-init.d` + `services.d`, NOT `s6-rc.d` bundles — do not introduce s6-rc).

`services.d/detector/run`: shebang `#!/usr/bin/with-contenv bashio`. Export `PYTHONPATH=/opt/argus/detector` (detector source is copied there by the Dockerfile), log a start line via `bashio::log.info`, then `exec python3 -m argus_detector.server`. The detector reads ARGUS_GRPC_BIND (127.0.0.1 in local mode) and ARGUS_MODEL_ROOT (/data/models) from the s6 container environment that config-gen wrote in Phase 1 — do not re-set them here. In remote mode this script is never started because config-gen writes the `down` file (Task 4); no in-script mode check is needed.

`services.d/orchestrator/run`: shebang `#!/usr/bin/with-contenv bashio`. Read the mode from `/run/argus/mode` (written by config-gen). If mode is `local`, log "waiting for detector" and run `python3 /usr/local/bin/wait-detector.py 127.0.0.1:50051 argus.v1.DetectorService` (blocks until SERVING). If mode is `remote`, skip the poll (the orchestrator dials the external endpoint itself). Then `exec dotnet /opt/argus/orchestrator/Argus.Orchestrator.dll`. Note in a comment that the orchestrator additionally enforces an in-app readiness gate (DetectionGateway.WaitForHealthyAsync) — the run-script poll is the belt-and-suspenders ordering guarantee per ARCHITECTURE.md.

Both `finish` scripts: shebang `#!/usr/bin/with-contenv bashio`. s6 passes the exit code as `$1`. Log a fatal line (`bashio::log.fatal`) reporting the service exited, then `exec /run/s6/basedir/bin/halt` — the s6-overlay v3 halt form (NOT the v2 `s6-svscanctl -t` form, which has no effect on v3 base images per PITFALLS.md). This makes a dying service bring the whole container down (PROC-03) so HA Supervisor reports the failure instead of a silent restart loop. Do not echo or log any credential or token value.

Set the execute bit on all four scripts via git so the bits survive checkout (`git update-index --chmod=+x` on each path) — Windows git strips execute bits and s6 will refuse to run non-executable service scripts.
